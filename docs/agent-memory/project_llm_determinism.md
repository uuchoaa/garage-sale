Example-app screens (Cashflow, Planetaria) are intended to be generated by an LLM from structured briefings, not hand-authored long-term. wise-ui's API is therefore designed against a determinism + auditability bar: the same brief should yield equivalent code across runs, across models, and across sessions.

**Implications for API design:**
- **Closed token vocabularies.** Props like `tone`, `size`, `weight`, `gap` take values from small exported unions (`Tone`, `Size`, `Gap`). LLM picks from an enum; it never invents values.
- **Zero Tailwind at the consumer layer.** `class="grid grid-cols-..."`, `class="ml-auto"`, etc. do not appear in screen code. Layout is expressed via DS primitives (`<Grid>`, `<Stack>`, `<Cluster>`) with token props. Exceptions are narrowly scoped (e.g. an `sr-only` replacement lives in the DS as `<VisuallyHidden>`).
- **One canonical way per concept.** No duplicates (`Menu` and `OverflowMenu` became one `Menu`). No optional aliases.
- **Fixed slot vocabulary.** Slot names (`#brand`, `#actions`, `#nav`, `#rail`, `#trigger`, `#leading`, `#title`, `#meta`, `#trailing`, `#header`, `#body`) are stable and documented.
- **Fixed page scaffold.** Every screen starts with `<Shell><Page><PageHeading>…`. Deviations are fail-states.
- **Typed briefing → typed screen.** Long term, a briefing schema (structured doc) maps mechanically to screen code; screen-level snapshot tests catch drift.
- **Docs-as-prompt.** The DS docs are the implicit prompt the LLM consumes. They must be enumerative and assertive — no "usually", "you can also", "it depends".

**Why:** Non-deterministic consumer code makes LLM regeneration expensive and reviews unreliable. If the same brief produces three different layouts, we lose the ability to diff, snapshot, and audit. A tight DS vocabulary is the enabler.

**How to apply:**
- When surfacing a new API, ask: is there exactly one way to express this? If not, pick one and eliminate the alternative.
- Avoid optional-but-useful flexibility. Flexibility is the enemy here.
- When a Tailwind class appears in consumer code, treat it as a DS gap, not a shortcut.
- Document primitives enumeratively with tables of valid values, not prose.
